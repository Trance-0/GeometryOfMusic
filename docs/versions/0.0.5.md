# 0.0.5

**Date:** 2026-04-20.

Playback visibility + timeline edit robustness round. Six bugs /
refinements called out together; each fix documented below so regression
checks have a named owner.

## Multi-dyad highlight during playback

Previously `playStep` only glowed the last track's dyad — multiple
tracks sounding at the same cell left only one node bright.

- New `TorusView.highlightDyads(dyads)` highlights a set of nodes;
  `highlightDyad(dyad)` becomes a thin wrapper.
- `playStep` collects every placement whose span covers the current
  cell (including chords held from earlier cells), so all sounding
  tracks light up simultaneously.

## Closed track curves

`CatmullRomCurve3(pts, closed=false, "centripetal")` →
`CatmullRomCurve3(pts, closed=true, "centripetal")`. Each track's
progression is drawn as a closed loop connecting the last placed chord
back to the first, matching the looping playback in the scheduler.

## Red progress indicator

New `TorusView.setPlayhead(dyads)` renders a bright red marker at every
currently-sounding dyad, plus a red line connecting them when two or
more tracks are active (a triangle when all three play at the same
step). Hidden when transport is stopped.

## Track-colored nodes on each trajectory

Nodes on a track's curve now take the track's color instead of the
interval-class color. If a dyad is shared between tracks, the
first-listed track wins (Lead beats Bass beats Pad). Unused nodes
still fall back to their interval-class color. The emissive glow for
the highlight still uses the interval-class color, so the two reads
remain distinguishable.

## Preset dropdown sticks to the loaded preset

The dropdown now displays the loaded preset's name until the user
edits the timeline, at which point it reverts to the
"— Starter templates —" placeholder. Implementation: `state.currentPresetId`
is set when a preset loads; `refreshPaths` clears it on any
placement-level change, guarded by `state.loadingPreset` so the load
itself doesn't immediately self-clear.

## Drag-to-extend no longer deletes chords

Previously the drag grip set pointer capture on itself and called
`rebuildTrack` on every accepted move — which destroyed the capturing
element and dropped the drag mid-motion, so the chord appeared to get
stuck, shortened, or disappear. The drag is now listened on the
document itself, so rebuilds of the track's cell container don't
break it. Duration is still clamped to `[1, maxDurationAt]`: the
chord "fits to the maximal empty space" when dragged too far right
and "fits to the shortest note" (one cell) when dragged too far left,
never zero and never overlapping.

## Mute button fixed + icon

- Closure-freshness bug: both the mute click handler and the
  instrument-selector change handler read from the closure-captured
  `track` object, which became stale whenever `rebuild()` replaced
  the settings record. Second clicks flipped the wrong value. Both
  now read `this.settings.tracks[t]` live.
- Text `mute` / `unmute` swapped for an SVG speaker icon (muted
  variant has an X on the speaker), sized to fit in the track header
  at a fixed 24×22 pixel footprint. The instrument selector was
  narrowed from 88px to 72px to make room.
